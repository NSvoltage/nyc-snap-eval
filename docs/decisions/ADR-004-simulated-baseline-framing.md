# ADR-004: Simulated baseline framing

**Status:** Accepted
**Date:** 2026-05-11
**Decision-makers:** [project lead]

## Context

A central part of the eval suite's value is showing the delta between a poorly-scaffolded and a well-scaffolded benefits-AI system. The natural comparison is to the MyCity chatbot — but we don't have access to NYC's MyCity system (it was taken down in February 2026), and we don't have its system prompt, retrieval configuration, or model version.

The risk is appearing to claim we have reproduced MyCity, or appearing to dunk on the specific vendors (Microsoft, Nuvalence) and agencies (NYC OTI, SBS) involved.

## Decision

We include a "simulated baseline" in the eval comparison: a deliberately under-scaffolded configuration consistent with publicly documented failure patterns. Specifically:
- Same model class as the reference implementation (Claude Sonnet 4.5)
- Weak system prompt without statutory grounding or refusal patterns
- RAG-light: naive retrieval over public SNAP documents with no statutory hierarchy
- No tool calls — eligibility statements are generated by the LLM
- No citation enforcement

Every reference to this baseline in any public-facing document — README, methodology PDF, blog post, results tables, ADRs — explicitly labels it as "simulated baseline" or "simulation, not replication." It is never referred to as MyCity or as a reproduction of any deployed system. The methodology PDF includes one paragraph naming what the simulation does and does not represent.

## Consequences

**Positive:**
- Gives us a meaningful delta to report ("the eval suite can distinguish a well-scaffolded system from a poorly-scaffolded one") without making claims we can't support
- Avoids the appearance of dunking on specific vendors
- Honest about what we don't know about MyCity's actual configuration

**Negative:**
- Less viscerally compelling than "here's MyCity-style numbers vs. ours"
- Requires careful framing in every document; the labeling discipline is non-negotiable

## Alternatives considered

- **Skip the baseline, report absolute numbers only.** Considered. Rejected because the delta is the most informative signal we have for an eval.
- **Reproduce MyCity from public reporting.** Rejected. We don't have enough technical detail to reproduce it faithfully, and the claim "this is MyCity" is one we cannot defend.
- **Use bare Claude Sonnet 4.5 with no scaffolding as the baseline.** Considered. Partially adopted: bare Sonnet 4.5 is one of several baselines we report. The "simulated MyCity-style" baseline is a separately labeled configuration that goes further — weak prompt, no tools, no citation enforcement — and the delta between it and the reference implementation is what we report most prominently.

## The labeling discipline

In any document, any chart, any results table, the baseline appears as one of:
- "Simulated baseline"
- "Simulated under-scaffolded configuration"
- "Baseline (simulation, not replication)"

It never appears as:
- "MyCity"
- "MyCity reproduction"
- "Pre-fix configuration"
- Any vendor or agency name

When results tables show the baseline, the table caption or surrounding paragraph includes the disclaimer: "The simulated baseline is a deliberately under-scaffolded configuration consistent with publicly documented failure patterns. It is not a reproduction of any specific deployed system."
