# ADR-002: Eligibility math is deterministic, not generative

**Status:** Accepted
**Date:** 2026-05-11
**Decision-makers:** [project lead]

## Context

The MyCity failure was specifically about confident, illegal-action-inducing model output on questions that have unambiguous answers in statute. The failure mode the reference implementation has to avoid is having the LLM compute or sample eligibility determinations that should be derived from rules.

Public benefits administration is a high-impact AI deployment surface per OMB M-25-21; the principal-basis test matters. Letting an LLM generate eligibility math means the LLM is the principal basis. Letting an LLM narrate eligibility math computed by a rules engine means the rules engine is the principal basis and the LLM is an interface.

## Decision

Eligibility computations in the reference implementation are never generated by the LLM. They are computed by a deterministic tool call. The LLM's role is to translate the user's question into structured input, call the eligibility tool, and narrate the result with citations.

Specifically:
- The reference implementation exposes a `check_eligibility(household_json)` tool that wraps the NYC Benefits Screening API
- The LLM may not return a categorical eligibility determination ("you qualify" / "you don't qualify") that was not produced by the tool
- The LLM may narrate, explain, surface caveats, and identify edge cases — but the determination itself is the tool's output
- When the tool's output is ambiguous or the user's facts are under-specified, the LLM surfaces the ambiguity and refuses to assert a determination

## Consequences

**Positive:**
- Eliminates the largest class of MyCity-style failures: confident wrong eligibility statements
- Aligns the architecture with OMB M-25-21's principal-basis framing
- Makes the implementation auditable: the eligibility logic is published Drools rules at `github.com/NYCOpportunity/ACCESS-NYC-Rules`
- Tests in Category B (factual recall) and Category D (eligibility edge cases) become naturally well-defined: did the LLM call the tool, and did it narrate the result faithfully?

**Negative:**
- The Benefits Screening API covers 35+ programs but is not exhaustive; some edge cases fall outside its coverage
- We are coupled to the API's behavior; if NYC takes it offline, we need a fallback (a deterministic Python port using OTDA tables)
- Cases that test the LLM's "raw" SNAP knowledge are now testing the tool integration, which is correct but worth flagging in the methodology

## Alternatives considered

- **Let the LLM compute eligibility from policy text + retrieved corpus.** Rejected. This is what MyCity effectively did. Even with high-quality retrieval, current models hallucinate on legal-domain factual questions at rates of 17–33% (Stanford RegLab on commercial legal-AI tools with RAG).
- **Use a custom rules engine.** Rejected. The NYC Benefits Screening API is already a deterministic Drools-backed rules engine maintained by NYC Opportunity. Building our own is scope creep and would compete with a public service.
- **Hybrid: LLM computes, tool verifies.** Rejected. Adds latency, adds a class of "what if they disagree" failures we'd have to test, and inverts the principal-basis relationship.
